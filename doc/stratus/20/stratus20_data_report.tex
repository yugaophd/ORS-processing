\subsection{Time Correction}
\begin{itemize}
\item Adjust time data to account for MATLAB's year zero issue.
\item Convert time units to 'days since 0001-01-01'.
\item Subtract 365 days from converted times to correct for MATLAB's leap year assumption.
\item Time base Interpolation (if necessary): Data are interpolated from the initial timing mark to the final timing mark, 
so this needs to be done before truncation.
\end{itemize}

Potential timing errors are addressed by referring to the "recover.xls" spreadsheet on buoy5, 
under the "subsurface" tab, to compare spike start and end times with the apparent time in the instrument data. 
If the offset is greater than one sample interval (+/- 5 min for Stratus deployments), 
it indicates misalignment in time, and the data is linearly interpolated to a new time base to synchronize instruments.

\subsection{Re-calculate Salinity}
\ifthenelse{\equal{\pressureavailable}{yes}}{
Pressure data is available for this deployment but may have drifted significantly 
in the \FirstInstrumentType{} instruments,
which can lead to inaccurate salinity calculations.
}{
Pressure data is not available for this deployment,
}
Therefore, we recalculate salinity using a fixed pressure value based on the instrument depth.
For \projectname{} \casenumber{}, salinity was recalculated using the following procedure:

\begin{enumerate}
    \item \textbf{Pressure Calculation:} Instead of using the potentially drifting pressure sensor data, we calculate a fixed pressure value based on instrument depth:
    \begin{itemize}
        \item Calculate pressure using the GSW function \texttt{p\_from\_z}, which converts geometric height to pressure
        \item Use the formula $z = -1 \times \text{instrument\_depth}$ (negative because depth is positive downward)
        \item Use latitude from the anchor survey for the calculation
    \end{itemize}
    
    \item \textbf{Practical Salinity Calculation:}
    \begin{itemize}
        \item Convert conductivity from S/m to mS/cm by multiplying with a factor of 10
        \item Apply the TEOS-10 (Thermodynamic Equation Of Seawater - 2010) algorithm via \texttt{gsw.SP\_from\_C}
        \item Use measured temperature and the calculated fixed pressure (rather than varying measured pressure)
        \item Store the computed values as \texttt{sea\_water\_practical\_salinity} with units 'psu'
    \end{itemize}
    
    \item \textbf{Absolute Salinity Calculation:}
    \begin{itemize}
        \item Use the TEOS-10 algorithm \texttt{gsw.SA\_from\_SP} to convert practical salinity to absolute salinity
        \item Incorporate geographical dependencies using latitude and longitude from the anchor survey
        \item Store the computed values as \texttt{sea\_water\_absolute\_salinity} with units 'g/kg'
    \end{itemize}
    
    \item \textbf{Post-Processing:}
    \begin{itemize}
        \item The original salinity variables (\texttt{sal} and \texttt{abssal}) are removed from the output datasets
    \end{itemize}
\end{enumerate}

This method ensures consistent salinity calculations and eliminates potential errors from pressure sensor drift, 
which can be substantial at deep ocean deployments. 
Using a fixed pressure calculated from the known instrument depth produces more reliable salinity values.

\section{Quality Control and Deployment Catalog}
The following quality control measures are implemented:

\subsection{Spike Removal}
Spike detection and removal are set up in function \texttt{src/qc\_function.py}. 
The spike removal function identifies and removes spikes from time series data using a rolling mean and standard deviation. 
It calculates these statistics within a specified window size around each data point. 
Points that deviate from the rolling mean by more than \ThresholdValue{} times the standard deviation are considered spikes and replaced with missing values (-99999). 
For \projectname{} \casenumber{} processing, the window size was \WindowSize{} points and the threshold was \ThresholdValue{} times the standard deviation.

\section{Merged Data Set}

\subsection{Metadata and Selection Strategy}
\begin{itemize}
    \item All necessary metadata are incorporated from individual deployments and retained in the merged file.
    \item Selection strategy for merging:
    \begin{itemize}
        \item If both sensors for a deployment have full records and are of good quality, 
        a Human-In-The-Loop (HITL) evaluation is done for the transitions at the start and end of the deployment, 
        and data records with the least discrepancy between deployments are chosen to be merged. 
        \item If one sensor is obviously bad or does not have a complete record, the other sensor is used.
        \item For \projectname{} \casenumber{}, \CurrentDeploymentInstrument{} was selected for the merged dataset.
        \item If the sampling rate of the two time series is different, the time series with the higher sampling rate is resampled to match the lower sampling rate.
    \end{itemize}
\end{itemize}

\section{Conclusion}
This report documents the preprocessing, quality control, and analysis steps 
for the deep temperature and salinity data from \projectname{} \casenumber{}. 
The data has been processed through several stages:

\begin{itemize}
    \item Time correction and data truncation to isolate the deployment period
    \item Quality control including spike removal and difference statistics
    \item Recalculation of salinity using the TEOS-10 algorithm at the instrument depth with observed temperature and conductivity
    \item Merging with adjacent deployment data % where overlap exists
\end{itemize}

The resulting cleaned and merged dataset provides a valuable record of deep ocean temperature and salinity 
at the \projectname{} site, contributing to long-term climate monitoring efforts.
